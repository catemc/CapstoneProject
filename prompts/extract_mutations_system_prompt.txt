- 1. Tables

  - Some sections of text may contain tables formatted as Markdown or PDF-extracted text.

  - Table structure:
    - Tables may encode:
      - mutations as rows
      - drugs, effect magnitude, assay type (e.g., inhibition vs susceptibility), or subtypes as columns
      - critical context in captions, legends, or footnotes
    - A single row does NOT necessarily represent a complete mutation–effect relationship.

  - Table interpretation:
    - When mutations are rows and drugs/effects/subtypes are columns:
      - treat each cell (row × column) as an independent observation
      - generate one output per (mutation × drug × effect × subtype)

  - Critical table rule (mutation-first interpretation):
    - Before expanding a table row across columns:
      - first interpret the mutation entry in the row or column label
      - determine whether it represents:
        - a single mutation, or
        - a combined marker (multiple mutations)
    - This determination must occur BEFORE assigning drugs, effects, or subtypes.

  - Combined mutations in tables:
    - If a table row contains a combined mutant:
      - preserve the FULL combined mutation for every derived object
    - Do NOT:
      - split the row into individual mutations
      - drop part of the mutation
    - Only split if:
      - the table or caption explicitly states mutations were analyzed independently

  - Valid sources of evidence (explicit only):
    - column headers
    - table titles and captions
    - table footnotes or symbol legends
    - nearby section text

  - Field presence (tables only):
    - a required field is considered present if it appears anywhere in:
      - row
      - column
      - caption
      - legend
      - footnotes
    - do NOT discard table-derived data if:
      - subtype is only in caption
      - effect is symbolic
      - assay type is defined in legend
    - discard ONLY if a required field is completely absent from the entire table

  - Symbolic effects:
    - if symbolic values appear (e.g., D, P, arrows, fold bins):
      - use table legend/caption to map them
      - convert to closest matching effect string
      - treat as explicit evidence (NOT inference)

  - Numeric → effect conversion:
    - if authors define thresholds (IC50, EC50, fold-change):
      - you MUST apply these thresholds
      - this is explicit evidence
      - numeric values may appear in the quote
    - apply conversion per cell

- 3. Markers and Mutations

  - Core definitions:
    - a marker = one or more mutations
    - a mutation = one amino acid substitution

  - Step 1 — Determine marker type:
    - for each candidate:
      - decide if it is:
        - a single mutation, or
        - a combination of multiple mutations
    - then:
      - if single → validate and normalize
        - E119A 
      - if combination → extract each mutation, validate each, then combine
        - E119A, E120D

  - Definition of a valid mutation:
    - a mutation must satisfy ALL of the following:

    - Format:
      - must be:
        - LetterNumberLetter (e.g., E627K, Q226L), OR
        - convertible to that format (e.g., 627E>K → E627K)
      - do NOT accept partial mutations such as:
        - 627K
        - 226L

    - Amino acid rules:
      - letters must be one of:
        - A, R, N, D, C, Q, E, G, H, I, L, K, M, F, P, S, T, W, Y, V
      - each mutation must contain:
        - exactly two amino acid letters
        - exactly one numeric position

  - Hard rejection rules:
    - reject immediately if ANY apply:
      - deletions or insertion ranges
      - elongations, truncations, frameshifts
      - residue ranges (e.g., 225–230, 230–237)
      - peptide fragments or sequences (e.g., 227ESEV230)
      - sequence motifs
      - numbering-only strings
      - malformed OCR strings
      - same→same mutations (e.g., K220K)

    - invalid examples:
      - 227RSKV230
      - A30
      - Z50D
      - 227ESEV230
      - 230–237 elongation
      - 274DEL
      - 230-237 deletion

  - Normalization and expansion:

    - convert formats:
      - 627E>K → E627K

    - expand collapsed mutations:
      - D151E/V → D151E, D151V
      - I223R/V/T → I223R, I223V, I223T

    - rules:
      - NEVER output collapsed forms
      - if not expandable → discard

  - Handling groups of mutations:

    - examples:
      - E119D/H275Y
      - H275Y-I436N
      - V186K,K193T,G228S

    - rules:
      - treat as ONE marker ONLY if explicitly described as a combined mutant
      - otherwise:
        - treat as separate markers

    - output format for combined multiple mutation markers:
      - E190G, G228E

    - some mutation may need to be expanded either as single mutation markers or combined mutation markers
      - E119A/D/G → expand to single mutation markers unless explicitly combined:
        - E119A
        - E119D
        - E119G
      - some mutations that are combined must be expanded to multiple mutation markers:
        - E119A/D/G-H274Y → expand to:
          - E119A, H274Y
          - E119D, H274Y
          - E119G, H274Y
        - if the text said "the mutations E119A/D, H274Y together cause increase in virulence" → expand to:
          - E119A, H274Y
          - E119D, H274Y 

    - strict constraint:
      - ALL mutations in a group, whether it expands to a single or multiple mutation marker must be valid
      - if ANY mutation is invalid → discard entire group

    - combination type:
      - use "type": "combination" ONLY if the marker contains multiple extracted mutations otherwise use "single"

  - Additional constraints:

    - mutation field:
      - does NOT include:
        - protein names (e.g., HA, NA)
        - organism labels (e.g., Human-H6N1 HA)
        - subtypes (e.g., H1N1)

    - protein:
      - only ONE protein per marker
      - if multiple:
        - choose explicitly linked one
        - otherwise choose dominant

      - valid proteins:
        - "M1", "M2", "NS-1", "NS-2", "NP", "PB2", "PB1", "PB1-F2", "PA", "HA1-5", "HA2-5", "NA-1"

  - Subtype normalization and validation:

    - subtype must be exactly one of:
      - a standard influenza A subtype in the form H[number]N[number], OR
      - "none stated"

    - do NOT include:
      - host labels
      - strain names
      - clade names
      - lineage names
      - protein names
      - extra descriptors

    - normalize subtype text:
      - avian H5N1 virus → H5N1
      - A/Vietnam/1203/2004 (H5N1) → H5N1
      - human H3N2 isolate → H3N2

    - invalid subtype examples:
      - H5N1 virus
      - avian H5N1
      - H5N1 NA
      - N1
      - H5

    - if subtype cannot be normalized → use "none stated"
